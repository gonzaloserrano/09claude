## Topic

A practical migration tip: treat Opus 4.7 as a different operating mode and use explicit migration logic for model names, prompts, and effort settings instead of only swapping the model string.

## Research Summary

- Anthropic’s new Opus 4.7 release and prompt-engineering guidance support the underlying claim that model upgrades sometimes need workflow-level adjustments.
- The “migrate to Opus 4.7” instruction is useful because it bundles config, prompting, and effort tuning together instead of pretending models are interchangeable.
- This is a good example of moving migration knowledge into a repeatable tool or skill.

## Key Findings

- Model migration is often about defaults, prompting style, and verification loops, not just the selected model ID.
- A dedicated migration skill reduces config drift across sessions and teammates.
- The core insight is operational: upgrade workflows, not just models.

## Practical Relevance (for Gonzalo)

- Useful if Gonzalo upgrades models often and wants repeatable behavior instead of manual config edits.
- The migration-skill idea can be reused for any toolchain change with lots of hidden defaults.

## Risks / Caveats

- A migration skill can lag behind the model if release assumptions change quickly.
- Over-fitting prompts to one model version can make later upgrades harder.

## Recommended Next Steps

1. Keep model migration logic in one place instead of scattered settings files.
2. Re-test effort, permission, and verification defaults after every major model upgrade.
3. Document what changed so regressions are easier to diagnose.
