## Topic

A workflow and measurement argument that AI-era engineering output should be tracked using normalized logical code change, not raw lines of code.

## Research Summary

- The GStack repository and README directly support the tweet’s emphasis on logical SLOC and role-based Claude Code workflows.
- Independent coverage frames GStack as a structured operational stack rather than just a prompt pack, which makes the productivity claim more than pure bravado.
- The measurement idea is the most durable part here: AI changes raw LOC so much that normalized output metrics become more useful.

## Key Findings

- Raw line counts are badly distorted by AI-assisted coding, so logical or normalized change is a more defensible productivity signal.
- GStack is part workflow system and part management philosophy layered on top of Claude Code.
- The strongest takeaway is methodological: measure shipped outcome and validated code change, not keystrokes.

## Risks / Caveats

- Any self-reported productivity framework can become branding if the methodology is not reproducible.
- Logical SLOC is better than raw LOC, but still not a full substitute for user or business outcomes.

## Recommended Next Steps

1. Track output with a small set of metrics: shipped changes, verified tests, and useful code delta.
2. Borrow the metric idea even if not adopting the full GStack stack.
3. Avoid comparing AI productivity claims without normalizing for task type and verification quality.
