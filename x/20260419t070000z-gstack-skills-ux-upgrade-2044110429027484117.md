## Topic

A GStack update focused less on raw capability and more on making skill usage easier to understand, invoke, and compare, drawing explicitly on classic UX principles.

## Research Summary

- The GStack repository and skills docs show a large command surface where usability matters as much as raw functionality.
- Independent commentary about combining skill stacks supports the idea that AI tooling is now complex enough to need serious information architecture and UX work.
- This tweet is really about the maturity shift from “collect more skills” to “make the skill system understandable.”

## Key Findings

- As skill libraries grow, discoverability and clarity become product problems, not just documentation problems.
- Borrowing from Steve Krug style UX principles makes sense because AI tooling often overwhelms users with hidden affordances.
- Better UX around skills can increase real usage more than shipping yet another command.

## Practical Relevance (for Gonzalo)

- Useful if Gonzalo accumulates many skills and starts feeling the management burden more than the capability gap.
- The broader lesson is to invest in discovery and naming before a skill library becomes a junk drawer.

## Risks / Caveats

- UX polish can hide underlying complexity without actually simplifying it.
- Large command systems still risk becoming brand-heavy and context-hungry.

## Recommended Next Steps

1. Audit any local skill set for discoverability and duplication.
2. Prefer a smaller set of high-leverage commands with obvious names.
3. Treat documentation, examples, and side-by-side previews as part of the product.
